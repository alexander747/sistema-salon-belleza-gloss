# Database & Migrations Specification

## Purpose

Define the MySQL database setup with TypeORM migrations, seed scripts, and Docker Compose infrastructure.

## Requirements

### Requirement: TypeORM DataSource

The system MUST use a TypeORM `DataSource` configured for MySQL 8 with environment-based configuration (`DB_HOST`, `DB_PORT`, `DB_USERNAME`, `DB_PASSWORD`, `DB_DATABASE`).

#### Scenario: DataSource connects to MySQL

- GIVEN a running MySQL 8 instance with correct credentials
- WHEN `DataSource.initialize()` runs
- THEN the connection succeeds and tables can be queried

#### Scenario: Connection fails with bad config

- GIVEN incorrect database credentials
- WHEN `DataSource.initialize()` runs
- THEN an error is thrown with a descriptive message

### Requirement: Migration-Only Schema Sync

`synchronize: false` MUST be set in the TypeORM config. All schema changes MUST go through migration files generated by the TypeORM CLI.

#### Scenario: synchronize is disabled

- GIVEN the TypeORM DataSource config
- WHEN inspecting `synchronize`
- THEN it is `false`

#### Scenario: Migration creates tables

- GIVEN an initial migration file
- WHEN `typeorm migration:run` executes
- THEN all entity tables are created with correct columns and foreign keys

#### Scenario: Migration rollback works

- GIVEN a migration that was just applied
- WHEN `typeorm migration:revert` runs
- THEN the migration is rolled back and the previous schema state is restored

### Requirement: Migration CLI

Migration files MUST be generated via `typeorm migration:create` or `typeorm migration:generate`. Migration source files MUST be in `src/database/migrations/`.

#### Scenario: Migration generation detects entity change

- GIVEN a new field added to an entity
- WHEN `typeorm migration:generate` runs
- THEN a new migration file is created with the ALTER TABLE statement

#### Scenario: Migration file naming

- GIVEN a generated migration
- WHEN inspecting its filename
- THEN it follows the pattern `TIMESTAMP-Description.ts`

### Requirement: Seed Script

The system MUST have a seed script (`src/database/seed.ts`) that creates: a SUPERADMIN user (email: admin@posfinal.app), and the default Rol values.

#### Scenario: Seed creates superadmin

- GIVEN an empty database
- WHEN the seed script runs
- THEN a superadmin user exists with email `admin@posfinal.app` and hashed password

#### Scenario: Seed is idempotent

- GIVEN a database with existing superadmin
- WHEN the seed script runs again
- THEN no duplicate user is created (upsert or skip)

### Requirement: Docker Compose

A `docker-compose.yml` at the project root MUST define two services: `mysql` (MySQL 8, port 3306, named volume for persistence) and `n8n` (n8n image, port 5678, with MySQL as database backend).

#### Scenario: Docker compose starts MySQL

- GIVEN Docker is installed
- WHEN `docker compose up -d` runs
- THEN the MySQL 8 container starts with the configured database

#### Scenario: n8n service is available

- GIVEN `docker compose up -d` has run
- WHEN curling `http://localhost:5678`
- THEN n8n responds with its health check or login page
